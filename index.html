How it works

Three retrieval signals composed into one query. A feedback loop that makes every signal better over time.

Every layer is independently useful. You can use vector search without the graph, the graph without rules, or rules without feedback. But the full pipeline is where compound improvement happens — retrieval gets better the more your agents use it.

Thompson Sampling

Beta-Bernoulli on every edge. Accept a result → edge gets stronger. Reject → weaker. Natural exploration/exploitation balance. The graph learns which paths matter.

Graph Engine

InMemory for dev, Memgraph for prod. Typed edges (REQUIRES, REFINES, USES, SIMILAR_TO). Personalized PageRank for structural retrieval.

The integration story

qortex isn’t a prototype. It’s a tested, benchmarked piece of infrastructure that plugs into the ecosystem you already use.

Three layers of agent memory

The market is splitting into three different problems:

What users said
Conversational memory. Entity extraction from chat history. User preferences, cross-session continuity.
mem0’s territory
What agents think
Self-managed agent state. LLM calls to edit memory blocks. ~16,900 tokens per memory management cycle.
MemGPT/Letta · expensive
What the system knows
Structured concepts, typed relationships, explicit rules, and a feedback loop that learns which paths matter.
qortex · 0.02ms · 0 LLM calls
Static vector search
Embed, cosine, return top-k. Flat. Static. No structure. No learning. Day one = day one thousand.
ChromaDB / Pinecone / Qdrant

An agent can use all three simultaneously: mem0 to remember the user prefers Python examples, MemGPT to maintain a coherent persona, and qortex to know that mTLS is the right choice for service mesh authentication. These aren’t competing approaches. They’re different layers.

The feedback loop nobody else has

Every query generates a query_id. After the agent uses the results, you report feedback: which concepts were accepted, rejected, or ignored.

This feedback adjusts edge weights using Thompson sampling. Accepted results strengthen the connecting edges. Rejected results weaken them. Next time a similar query runs, retrieval automatically favors paths that worked.

After 50 queries with consistent feedback: strong edges converge to Beta(45,6). Weak paths suppress to Beta(3,42). The graph becomes tuned to your domain’s actual retrieval patterns. Static vector stores stay static forever.

Current limits

v0.5.2. The full pipeline works. These are the open problems.

Causal DAG

The knowledge graph captures correlations via edge types. Causal discovery (which edges are causes vs associations) is the next phase.

Credit assignment

Multiple active concepts contribute to a result. Isolating which edge was responsible for a good outcome requires causal inference. Coming.

PPR at scale

Personalized PageRank is the one bottleneck. Proven sublinear path exists but not yet implemented for graphs over 100K nodes.

Multi-tenant

Per-user context partitioning for learning is designed (issue #76) but not shipped. Currently single-tenant.

See the GitHub issues for the full roadmap. Causal DAG is issue #97. Production deployment (triple interface: GraphQL + REST + MCP) is issue #63.

FAQ

How is this different from ChromaDB / Pinecone / Qdrant?

Those are vector databases. qortex is a knowledge graph that sits on top of vector search. We use vectors for initial retrieval, then add structural expansion via typed edges and domain rule projection. The graph layer adds +22% precision at zero overhead. You can use qortex on top of any vector store you already have.

Does this need an API key?

No. The knowledge layer, graph traversal, feedback loop, and MCP server all work without external API calls. If you want LLM-powered ingestion (extracting concepts from docs automatically), you need an Anthropic key. But the core retrieval engine is entirely local.

What about mem0 or MemGPT?

Different layers. mem0 does conversational memory (what users said). MemGPT does agent state management (what agents think). qortex does domain knowledge (what the system knows). An agent can use all three simultaneously. They compose, not compete.

How does the TypeScript/cross-language story work?

TypeScript adapters (Mastra, LangChain.js, OpenClaw) talk to the Python engine over MCP stdio. 29 MCP tool calls complete in 3.94 seconds. Server spawn is ~400ms one-time. For agent workflows where each LLM call takes 500ms–5s, transport overhead is invisible. No WASM, no FFI, no rewrite.

How many feedback signals before the graph learns?

Edges start at Beta(1,1) — uniform prior. The first 10 signals make the biggest difference (uncertainty to reasonable confidence). By 50, the graph has a strong model. By 200, it’s highly tuned to your domain. The system is most useful where it’s deployed longest.

Is this production-ready?

Used in production via OpenClaw on every agent turn. 1,960 tests. v0.5.2 on PyPI. 7 framework adapters passing their own test suites. Full OpenTelemetry instrumentation. But causal DAG and multi-tenant partitioning are still open — see the limits above.

What data leaves my machine?

Nothing, by default. The knowledge graph, vector index, learning state, and feedback history all live in local SQLite at ~/.qortex/. LLM-powered ingestion sends prompts to Anthropic if you opt in. The core retrieval loop is entirely local.
