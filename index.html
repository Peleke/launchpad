What is buildlog?

Every AI coding session produces decisions, corrections, and outcomes. Almost all of it is discarded. The next session starts from scratch.

buildlog sits between you and your AI agent. It captures structured work trajectories, extracts recurring patterns as rules, and uses a contextual bandit to decide which rules to surface in your agent's instruction set. Then it measures whether that selection reduced mistakes.

The core loop is: capture → extract → select → render → measure → update.

journal entries → extraction pipeline → candidate rules
    ↓
Thompson Sampling (Beta-Bernoulli bandit, per-context posteriors)
    ↓
selected rules → render to CLAUDE.md / .cursorrules / copilot-instructions.md / ...
    ↓
experiment: log mistakes → compute RMR → update posteriors → loop

The buildlog stack

MCP Server

34 tools exposed via Model Context Protocol. Your agent calls buildlog during sessions: log commits, run gauntlet reviews, query bandit state, close feedback loops.

Thompson Sampling

Beta-Bernoulli contextual bandit. Error classes partition the state space. Seed rules start at Beta(3,1). Non-seed at Beta(1,1). ~20 observations per context for meaningful differentiation.

Extraction Pipeline

Regex (fast, free), OpenAI, Anthropic, or Ollama extractors. Semantic deduplication via sentence-transformers or OpenAI embeddings. Pluggable at every stage.

Review Gauntlet

Multi-persona code review system. Security, architecture, testing, performance reviewers. Runs on commits, categorizes by severity, files GitHub issues.

Experiment Framework

Start/stop sessions. Log mistakes with typed error classes. Compute Repeated Mistake Rate across sessions. Statistical measurement, not anecdotes.

Multi-Agent Render

Same knowledge base renders to Claude Code, Cursor, GitHub Copilot, Windsurf, Continue.dev, and VS Code. Switch agents without losing what you learned.

Global SQLite

Single database at ~/.buildlog/buildlog.db. WAL mode for concurrent access. Project isolation via SHA-256 hashed IDs. Legacy JSONL fallback for older installs.

Pluggable Backends

Builtin bandit or qortex for cross-consumer credit propagation and knowledge graph integration. LearningBackend protocol with two shipped adapters.

Agent targets

One knowledge base. Every agent format. buildlog promote --target <agent> writes learned rules to the file your agent reads.

Get started

Global mode (recommended)

One install, works in every project. Claude Code gets all 34 buildlog tools in every session.

$ uv tool install buildlog # or: pipx install buildlog
$ buildlog init-mcp --global -y

Per-project

$ uv pip install buildlog
$ buildlog init --defaults

With extras

$ uv pip install "buildlog[all]"
# or pick: [embeddings] [openai] [anthropic] [ollama] [qortex]

Verify

$ buildlog mcp-test # 34 tools registered
$ buildlog overview # project state

Python 3.10+. The qortex backend requires 3.11+.

Current limits

v0.15 ships the full capture-extract-select-render-measure loop. These are the open problems:

Extraction quality is uneven. Regex extractors miss nuance. LLM extractors are accurate but expensive. The middle ground is still being found.

Feedback signals are coarse. Repeated Mistake Rate works but requires manual tagging. Richer automatic signals (test outcomes, review results, revision distance) are on the roadmap.

Credit assignment is limited. When multiple rules are active, the system does not yet isolate which one was responsible for an outcome.

Single-agent only. Shared learning across multiple agents in the same session is designed but not implemented.

See the roadmap for what's next: embedding persistence via sqlite-vec, cross-project convergence, and emergent rule graphs.

FAQ

How is this different from editing CLAUDE.md by hand?

Manual editing is the starting point. buildlog adds the rest: extracting patterns from real work instead of guessing, selecting which rules to include based on observed outcomes instead of intuition, measuring whether they reduce mistakes, and demoting rules that don't. Manual curation does not self-correct.

Does this require an API key?

No. The base install uses regex extraction and local computation. LLM-backed extraction (OpenAI, Anthropic, Ollama) is available via optional extras for richer patterns, but the core loop runs without any external calls.

How many observations before the bandit is useful?

Seed rules start with boosted priors of Beta(3,1), reflecting the expectation that curated rules are more likely to be effective. With ~20 observations per context, you get meaningful differentiation. The system is deliberately conservative with limited data.

What data leaves my machine?

Nothing, by default. Everything is in ~/.buildlog/buildlog.db (local SQLite). If you enable LLM extraction via an API key, extraction prompts go to the provider you chose. The bandit, experiments, and core loop are entirely local.

Can I use this without Claude Code?

Yes. The MCP server is Claude Code-native, but rule rendering works for Cursor, Copilot, Windsurf, Continue.dev, and VS Code. The CLI works standalone: buildlog commit, buildlog experiment start, buildlog promote --target cursor.
