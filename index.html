Install it. Pick your adapter. Go.

qortex is a Python package. Add it as a dependency, select the adapter for your framework, and your retrieval layer now has a knowledge graph behind it. No API keys for the knowledge layer. No config files.

# As a backend for your framework
$ uv add qortex
$ pip install "qortex[all]" # embeddings + production graph + everything
# or pick: [vec] [vec-sqlite] [memgraph]

# As an MCP server (Claude Code, Cursor, Windsurf)
$ claude mcp add qortex -- uvx qortex mcp-serve
$ uvx qortex mcp-serve # any MCP client

Your retrieval layer now returns structurally related concepts (not just textually similar ones), applies domain rules, and persists everything to SQLite across restarts. Python 3.11+. macOS, Linux, Windows.

Why we built a different kind of backend

We were building agents. They were bad at remembering things. Not because the LLM was dumb, but because the retrieval layer was limited. The standard approach (embed, cosine, top-k) works well for simple lookups. But ask something that spans multiple concepts (“How do I implement enterprise SSO?” where SAML, OpenID Connect, and OAuth2 are all relevant but share zero vocabulary) and flat vector search misses the connections.

The agent frameworks themselves are excellent. CrewAI, LangChain, AutoGen, Agno, Mastra: they handle orchestration, tool use, planning, and a hundred other things well. But retrieval backends are a different problem. We wanted a backend that could follow typed relationships between concepts, apply domain rules, and learn from feedback over time. That backend did not exist, so we built it.

qortex is a knowledge graph where every edge carries a probability distribution. Accept a result, the path strengthens. Reject it, the path weakens. The graph remembers what worked and surfaces it first next time. We plugged it into every major framework as a drop-in adapter and ran the benchmarks to see if the architectural choices would show up in the numbers.

Metric qortex Framework default Your gain
Precision@5 0.55 0.45 +22%
Recall@5 0.81 0.65 +26%
nDCG@5 0.716 0.628 +14%
Added overhead graph traversal: 0.02ms free

20-concept authentication domain. The “Framework default” column is standard cosine similarity, the same retrieval every framework ships. Every number is reproducible.

On cross-cutting queries (the ones that span vocabulary boundaries, like “enterprise SSO” pulling in SAML + OAuth2 + OpenID Connect) the gap widens further: +50% precision and +71% ranking quality. That is where graph structure pays off most, because typed edges (REQUIRES, REFINES, USES) connect concepts that share zero vocabulary.

The numbers are a consequence of the architecture, not a trick. A knowledge graph with typed relationships and a feedback loop finds connections that flat vector search cannot. And because the feedback loop accumulates signal over time, accuracy increases the longer the system runs. That is why we built it.

How it works

Three retrieval signals composed into one query, and a feedback loop that makes every signal better over time.

In the authentication benchmark, composing all three signals raised precision from 0.45 to 0.55 and recall from 0.65 to 0.81. The feedback loop then tunes the weights between those signals based on what your agent actually accepts or rejects.

Current limits

v0.5.2 is running in production via OpenClaw. These features are not shipped yet.

Causal discovery

The graph captures relationships. Distinguishing cause from correlation is the next phase.

Credit assignment

When multiple concepts contribute to a good result, isolating which edge was responsible is hard. Working on it.

Large-scale PPR

Graph traversal is fast up to ~100K nodes. Beyond that, we need sublinear PPR. Not shipped yet.

Multi-tenant

Currently single-tenant. Per-user learning partitions are not shipped yet.

Full roadmap on GitHub.

FAQ

How is this different from ChromaDB / Pinecone / Qdrant?

Those are vector databases. qortex is a knowledge graph that sits on top of any of them. You keep your existing vector store for embedding and similarity search. qortex adds a second layer: structural traversal across typed relationships, domain rules, and a feedback loop. They are complementary, not competing.

Does this need an API key?

No. The knowledge graph, retrieval, feedback loop, and MCP server all work locally without any API calls. If you want to auto-extract concepts from docs (optional), you need an Anthropic key. But the core engine is entirely local and free.

What about mem0 or MemGPT?

Different layers entirely. mem0 remembers what users said. MemGPT manages what agents think. qortex knows what’s true about your domain. Use all three together; they compose.

I use TypeScript. Does this work?

Yes. Mastra and LangChain.js adapters talk to the Python engine over MCP. 29 tool calls in 3.9 seconds over real stdio. For agent workflows where each LLM call takes 500ms–5s, you won’t notice the transport.

How fast does the feedback loop kick in?

First 10 signals make the biggest difference. By 50, strong edges are at Beta(45,6) and weak ones at Beta(3,42). By 200, retrieval rankings are highly tuned to your domain. The graph accumulates signal over its entire lifetime, so accuracy increases monotonically with deployment duration.

Is this production-ready?

Running in production via OpenClaw on every agent turn. 1,960 tests. 7 framework adapters passing their own test suites. Full OpenTelemetry tracing. Causal DAG and multi-tenant are still open (see limits above).

What data leaves my machine?

Nothing, by default. Graph, vectors, learning state, and feedback history all live in local SQLite. If you opt into LLM-powered ingestion, prompts go to Anthropic. The core retrieval loop is entirely yours.
